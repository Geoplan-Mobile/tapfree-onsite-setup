# 05. PRM 게이트 / 영역 설정

← [04. Geospace 앵커 설정](./04-geospace-anchors.md)

기본 설치본에는 이미 **성남 캠퍼스 기준의 영역들** 이 추가되어 있다.
본 단계에서는 이 영역들의 **위치(좌표)만 현장값으로 이동/수정** 하면 된다. 신규 영역 생성 작업은 필요하지 않다.

> **전제**: 본 단계는 게이트의 **물리적 설치 및 위치 좌표 측정이 완료된 상태**에서 시작한다.

---

## 사전 준비물

- [ ] **게이트 폴리곤 목록** (현장 측량 완료) — 모든 게이트의 폴리곤이 미리 계산되어 있어야 한다.
  - 한 게이트의 **한 꼭짓점 좌표** 를 기준점으로 측정
  - 게이트의 **가로 / 세로 길이**
  - **게이트 간 간격**
  - 위 세 가지 값을 조합하여 **모든 게이트의 네 꼭짓점 좌표(폴리곤)** 를 산출해 둔다.
- [ ] [04. Geospace 앵커 설정](./04-geospace-anchors.md) 완료 (좌표계 기준이 동일해야 함)

## 1. PRM 페이지 접속

1. 브라우저에서 PRM 페이지 접속 — `http://{host}:7081/prm/login`
2. 계정 로그인 — `admin` / `admin`
3. **`test` 프로젝트** (기본 프로젝트) 선택

---

## 2. 게이트 위치 수정

> **전제**: 본 단계 진입 전, 모든 게이트의 **폴리곤 좌표** 가 산출되어 있어야 한다. (사전 준비물 참조)

1. **진출입 영역 설정 (Entry/Exit Area Settings)** 으로 이동
2. 이름이 `GATE_` 로 시작하는 영역들을 확인
3. 적당한 영역을 골라 측정된 **게이트 폴리곤 좌표대로 영역을 수정**

> **유의사항**
> - `GATE_*` 영역은 실제 진출입 영역을 그리기 위한 **reference** 및 **게이트 위치를 육안으로 확인** 하기 위한 용도이다.
> - **외부 연계 (External Linkage)** 를 반드시 **N** 으로 설정해야 한다.
> - 모바일에서는 본 영역의 진출입을 판단하지 않는다.

---

## 3. 영역 수정

작업 방식은 [2. 게이트 위치 수정](#2-게이트-위치-수정) 과 동일하며, 본 단계에서는 `FREE`, `PAID`, `AISLE_*` 영역을 편집한다.

### 편집 규칙

- **두 게이트와 그 사이의 내부 영역(통로)을 제외한 나머지 영역** 을 절반은 `FREE`, 절반은 `PAID` 로 설정한다.
- 이미 그려진 `GATE_*` 폴리곤을 reference 로 활용해 `FREE` / `PAID` 폴리곤을 완성한 후 저장.
- **AISLE 영역** 은 `GATE_*` 사이의 통로 영역을 절반으로 나누어:
  - `AISLE_1_1` → `FREE` 측에 위치
  - `AISLE_1_4` → `PAID` 측에 위치

#### `AISLE_*` 이름 규칙

`AISLE_<aisleId>_<n>` 형식으로, 두 숫자가 각각 다른 의미를 가진다.

| 부분 | 의미 |
|---|---|
| `<aisleId>` (첫 번째 숫자) | 통로 ID. 같은 통로의 영역들은 모두 이 값이 같다. <br>(예: `AISLE_1_1`, `AISLE_1_4` → 둘 다 통로 1) |
| `<n>` (두 번째 숫자) | 같은 통로 내 영역 구분 번호. `1` = `FREE` 측, `4` = `PAID` 측. |

> 이 `<aisleId>` 값은 [07. Tapfree-edge 설정](./07-tapfree-edge-config.md) 의 `aisles[].id` 와 매칭되어 모바일 / Edge 가 통로를 식별한다.

#### 모바일 라이브러리에서의 사용

- 영역 진입은 모바일 라이브러리의 **`onLocation`** 콜백을 통해 **`areaName`** 으로 전달된다. (예: `"AISLE_1_1"`)
- 이 중 `<aisleId>` 값은 게이트 연결 함수 **`connectPayload`** 의 `aisleId` 인자로 사용된다.

## 5. 영역 세부 조정

본 문서의 1 ~ 4 단계는 **아주 기본적인 영역 설정** 만 다룬다.

실제 운영에서는 [07. Tapfree-edge 설정](./07-tapfree-edge-config.md) 까지 완료한 뒤, **모바일에서 `onLocation` 이벤트를 직접 확인하면서 적절한 시점에 이벤트가 발생하도록 영역(폴리곤)을 미세 조정** 하는 작업이 추가로 필요하다.

> ⚠️ 본 단계의 미세 조정으로 PRM 영역이 변경되면 **Tapfree-edge 의 재시작** 이 필요하다. 변경된 area_info 가 부팅 시 로드되기 때문에 재시작 없이는 변경이 반영되지 않는다.

## Appendix — 영역 이름은 자유, 모바일 노출 여부는 외부 연계로 제어

본 문서의 영역 이름은 `GATE_*`, `FREE`, `PAID`, `AISLE_<aisleId>_<n>` 이라는 현재의 **운영 컨벤션** 을 기준으로 기술되어 있으나, **이 이름들은 시스템이 강제하지 않으며** 앱 측 개발자가 자유롭게 정의할 수 있다.

| 레이어 | 동작 / 제약 |
|---|---|
| tapfree-edge 서버 | - ❌ 규정된 영역 이름 없음. <br>- PRM 의 모든 inout 영역 중 **외부 연계 (External Linkage) = Y** 인 것만 모바일에 전달 한다. <br>- 따라서 모바일에서 영역이벤트가 필요하다 면 반드시 Y로 설정.|
| iOS / Android 라이브러리 | - ❌ PRM에 **외부 연계 = Y** 인 영역에 한해 `onLocation()` 으로 PRM 의 영역 이름을 **그대로 전달**. <br>- 통로 내 영역명은 앱이 직접 `aisleId`를 유추 할 수 있어야 한다. <br>- `aisleId` 는 **2 자리 문자열** 로 제한된다.<br>- 단, `connectPayload()` 는 aisleId가 1자리일 경우 내부적으로 `0` 을 패딩하여 사용한다. <br>- 이 **0-padded 2 자리 문자열** 이 [07. Tapfree-edge 설정](./07-tapfree-edge-config.md) 의 `aisles[].id` 와 일치해야 한다. |

### 모바일 노출 여부의 단일 제어 지점 — 외부 연계 (External Linkage)

영역을 모바일로 전달할지 / 차단할지는 **PRM 의 외부 연계 (External Linkage) 플래그 하나로 결정** 된다.

- **외부 연계 = Y** : 모바일로 전달되어 이벤트 발생시 `onLocation` 호출
- **외부 연계 = N** : tapfree-edge 가 모바일에 **전달 하지 않음**. 시각화 / reference 용 (예: `GATE_*`)등이 N으로 설정. 

> ⚠️ 시각화 / reference 용 영역 진출입 이벤트가 필요 없으므로 **외부 연계 = N** 으로 세팅해야 한다. <br>그렇지 않으면 모바일에 함께 전달되어 해당영역의 onLocation 이벤트가 발생 할 수 있다.
